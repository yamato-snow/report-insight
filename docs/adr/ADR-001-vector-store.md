# ADR-001: ベクトルストアに pgvector を採用する

| 項目 | 内容 |
|---|---|
| ステータス | 承認 |
| 日付 | 2026-07-17 |

## コンテキスト

RAG検索（F-2）のためにベクトルストアが必要。候補は pgvector（RDS PostgreSQL 拡張）、Amazon OpenSearch、Pinecone 等の専用サービス。

前提となる規模感：

- 報告書 400件/日 ≒ 15万件/年。1報告書≒1チャンクなので**数年運用しても100万ベクトル未満**
- 検索は社内管理者のみ（〜40名）、ピーク同時数件
- クライアントの運用体制は情シス2名（専任DBAなし）

## 決定

**RDS PostgreSQL + pgvector を採用する。**

## 理由

1. **規模が小さい**：pgvector + HNSW は100万ベクトル規模まで実用十分。本件はその1/10以下
2. **ハイブリッド検索が1クエリで書ける**：メタデータフィルタ（物件・期間・分類）と認可フィルタをベクトル検索と同一SQLで実行できる。専用DBだと二段検索や属性同期が必要になる
3. **運用コンポーネントを増やさない**：業務データがすでに PostgreSQL にある。バックアップ・監視・障害対応の対象が1つで済み、情シス2名の体制に合う
4. **コスト**：追加費用ゼロ（RDS費用に含まれる）。OpenSearch 最小構成でも月2〜3万円、Pinecone は従量＋データ主権の説明コストが発生

## 却下した選択肢

- **Amazon OpenSearch**：全文検索（形態素）も欲しくなった場合の将来候補。現時点ではオーバースペック
- **Pinecone 等 SaaS**：性能は魅力だが、社外秘データを追加ベンダーに預ける承認コストが本件の規模に見合わない

## 影響・再検討条件

- ベクトル数が100万を超える、または検索レイテンシ p95 が1秒を超えたら OpenSearch 移行を再検討
- 移行に備え、検索は `SearchRepository` インターフェースの背後に隔離する

## 追記（2026-07-19）

閾値までの余裕を本システム相当の構成で実測した。50万件・フィルタ併用の最悪ケースでも
p95 283ms（閾値1秒の約1/3.5）。詳細は [tools/benchmarks/pgvector/](../../tools/benchmarks/pgvector/README.md) を参照。
