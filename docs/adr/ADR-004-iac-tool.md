# ADR-004: IaC ツールに Terraform を採用する（SAM / CDK ではなく）

| 項目 | 内容 |
|---|---|
| ステータス | 承認 |
| 日付 | 2026-07-17 |

## コンテキスト

インフラ全体（VPC・ECS・RDS・S3・SQS・CloudWatch・IAM）をコード管理する。候補は Terraform、AWS SAM、AWS CDK、素の CloudFormation、Pulumi。

前提：

- ワークロードは **ECS Fargate＋RDS 中心のコンテナ構成**（[ADR-002](ADR-002-compute.md)）。Lambda 中心のサーバーレスではない
- 受託案件として、**引き渡し先（クライアント情シス2名）が保守できること**が要件
- 変更は「差分をレビューして承認してから適用」という受託の承認フローに乗せたい

## 決定

**Terraform を採用する。** 付随的に Lambda が必要になった場合も SAM を混ぜず Terraform に統一する（ツール分散の回避）。

## 理由

1. **ワークロード適合**：SAM は Lambda/API Gateway 中心のサーバーレス特化フレームワーク。VPC・ECS・RDS が主体の本構成では SAM の抽象化がほぼ効かず、素の CloudFormation を書くのと変わらなくなる
2. **plan による差分レビュー**：`terraform plan` の差分を PR コメントに出す運用が、受託の「承認してから適用」フローにそのまま載る（[IaC戦略 §5](../11_iac_strategy.md)）
3. **引き渡し先の保守性**：宣言的 HCL は読み手の学習コストが低く、採用実績・人材プール・情報量が最大。CDK は合成された CloudFormation のデバッグに開発者スキルを要求し、情シス2名体制に合わない
4. **ステート管理とドリフト検知**：state を持つことで「コード外で行われた手動変更」を機械検出できる（週次 plan。IaC戦略 §8）

## 却下した選択肢

- **AWS SAM**：本件では不適合だが、**Lambda 中心のイベント駆動構成（例：S3トリガの軽量処理だけの案件）なら第一候補**。ローカルテスト（`sam local`）とデプロイの手軽さは優れており、ワークロードが変われば結論も変わる——という適材適所の判断自体を本ADRの記録として残す
- **AWS CDK**：プログラミング言語で書ける表現力は魅力だが、上記3の理由で見送り。開発チームが内製・常駐で継続保守する案件なら再評価する
- **素の CloudFormation**：冗長で差分レビューがしづらい
- **Pulumi**：採用実績が相対的に少なく、引き渡し先の保守性で不利

## 再検討条件

- 構成が Lambda 中心に変わる場合（SAM を再評価）
- クライアント側に CDK 運用経験者が常駐する体制になった場合
