# ADR-003: LLMはタスク別にモデルを使い分け、単一プロバイダ＋抽象化層で運用する

| 項目 | 内容 |
|---|---|
| ステータス | 承認 |
| 日付 | 2026-07-17 |

## コンテキスト

LLM 利用箇所は4種（分類・条件抽出・RAG回答・月次報告書）。すべてを最上位モデルで処理するとコストが予算（月¥10万）を大きく超える。一方、分類精度90%・緊急度recall 95%という品質基準もある（[LLM設計書 §4](../05_llm_design.md)）。

## 決定

1. **タスク別使い分け**：大量・定型タスク（分類・条件抽出）は Haiku 級、文章品質が必要なタスク（RAG回答・月次報告書）は Sonnet 級
2. **プロバイダは単一（Claude API）に絞る**が、呼び出しは `LLMClient` 抽象化層に隔離し、モデルIDは設定値にする
3. **フォールバック**：特定モデル障害時は同プロバイダ内の代替モデルへ自動切替。プロバイダ全断時は縮退運転（SQS滞留＋メタデータ検索のみ）

## 理由

- **使い分けの根拠**：分類はスキーマ強制＋評価セットで品質担保できるため、モデル格を下げても基準を満たせる（満たせるかどうかは評価セットが判定する——「安いモデルで基準を満たすことを評価で証明する」のが本設計の骨子）。コスト差は約7倍
- **マルチプロバイダにしない理由**：プロンプトは実測ではプロバイダ間で互換性が低く、評価セットの維持コストが2倍になる。本件の可用性要件（業務時間内99%・夜間バッチ許容）に対して過剰
- **抽象化層を置く理由**：モデルの世代交代が速く（年数回）、モデルIDのハードコード散在は技術的負債になる。プロンプトバージョンとモデルIDを `report_analyses` に記録し、切替の影響を後から追跡可能にする

## 却下した選択肢

- **全タスク最上位モデル**：品質は上がるがコスト約5倍。予算超過であり、分類タスクでは評価上の差も小さい
- **セルフホスト（オープンウェイト）**：GPU運用は情シス2名体制で非現実的。データ主権要件が強まった場合の将来候補として記録のみ
- **マルチプロバイダのアクティブ・アクティブ**：上記の通り過剰

## 再検討条件

- 評価セットで Haiku 級が基準を割り続けた場合、該当タスクのみ Sonnet 級へ昇格（コスト再試算とセット）
- 月次コストが2ヶ月連続で¥90,000を超えたら縮退ラダー（LLM設計書 §5）を発動
