# ADR-002: コンピュートに ECS Fargate を採用する（Lambda ではなく）

| 項目 | 内容 |
|---|---|
| ステータス | 承認 |
| 日付 | 2026-07-17 |

## コンテキスト

API（FastAPI）と構造化ワーカーの実行基盤。候補は ECS Fargate、Lambda、EC2、App Runner。

ワークロード特性：

- worker：LLM呼び出しを含む処理で**1件あたり30秒〜数分**。日次バッチ後に集中し、レートリミットに合わせた同時実行制御（セマフォ）が必要
- API：SSEストリーミング（検索応答）を含む常駐型
- 社内システムのためトラフィックは平日日中に集中、スパイクは小さい

## 決定

**API・worker ともに ECS Fargate を採用する。**

## 理由

1. **実行時間とストリーミング**：Lambda の15分制限は月次報告書生成（複数LLM呼び出し）で危険域。SSE ストリーミングも ALB + 常駐プロセスが素直
2. **同時実行制御**：LLM のレートリミットに合わせた「プロセス内セマフォ＋SQSポーリング速度制御」は常駐ワーカーが実装しやすい。Lambda の同時実行制御は予約同時実行数の管理が別途必要
3. **コールドスタート回避**：業務時間中の検索応答3秒以内（非機能要件）に対し、コンテナ常駐で安定
4. **開発体験の一貫性**：ローカルの docker compose と本番のタスク定義が同型。受託の引き渡し先（情シス）にも説明しやすい

## 却下した選択肢

- **Lambda**：S3イベント起点の軽量処理なら最適だが、上記の実行時間・ストリーミング・同時実行制御で不利。「AI = とりあえずLambda」ではなくワークロードで選ぶ
- **EC2**：パッチ運用が発生。情シス2名体制で避けたい
- **App Runner**：APIには適するが worker（SQSコンシューマ）に不向きで、基盤が2種類になる

## コスト

dev は Fargate Spot + 最小タスクで月約$30。prod は api×2 + worker×1 で月約$120（RDS別）。

## 再検討条件

夜間・休日のアイドルコストが問題になったら、worker のみ EventBridge Scheduler での時間帯スケーリング（0タスク化）を導入する。
