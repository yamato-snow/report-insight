Report Insight — 画面モック MOCK

これは見た目と操作の確認用のモックです。本物のデータベースやAIは動いておらず、 ブラウザの中のダミーデータだけで動いています。承認をいただいた後に本体へ組み込みます。

使い方：右上の「利用者」を切り替えると、立場ごとに見える範囲が変わります。 何をしても壊れません。元に戻したいときは右上の「リセット」を押してください。

まずはここから

受入テスト チェックリスト

全29ケースを立場ごとに1件ずつ確認する表です。OK / NG をその場で記録でき、 途中でやめて再開もできます。終わったら結果をテキストで書き出せます。 まずこれを開いて、上から順に進めるのが早いです。

受入テスト用の治具

データの流れを検証する 検査用

報告書がどこを通って画面に出てくるかを可視化した画面です。 日次バッチを流すと、S3到着 → SQS → ワーカー処理 → DB保存 の各段の件数が進み、 緊急通知・未分類キューへの回送・処理失敗（DLQ）の内訳が出ます。 わざと異常を混ぜて、取りこぼしが起きないか・監視のしきい値に触れるかを確認できます。

報告を投入する 検査用

現場スタッフになったつもりで報告を書き、AIがどう仕分けるかをその場で見る画面です。 要件上、現場スタッフ向けの報告画面はスコープ外（既存システムを使い続ける）なので、 これは製品の機能ではなくテストのための入口です。 ここから投入すると下の「報告書」一覧に現れ、一連の流れを通しで確認できます。

今回追加する画面

報告書の一覧・確認

課題①「1日400件の確認負荷」に対応。一覧の絞り込みに加え、行をクリックして中身を読み、 AIが迷った報告書の分類を人が直せるようになりました。物件名の表示とページ送りも追加しています。

追加分：B（分類の直し）・C（詳細）・E（物件名）・F（ページ送り）

月次報告書

課題③「月次報告書に1物件2時間」に対応。物件と月を選ぶとドラフトが自動生成され、 編集 → 承認 → PDF出力まで画面だけで完結します。承認後は編集できなくなります。

追加分：A（月次報告書一式）

監査ログ

「誰がいつ分類を直したか」「誰が月次報告書を承認したか」の記録。 受入条件で「監査ログに残ること」が求められているため、確認できる場所を用意しました。
