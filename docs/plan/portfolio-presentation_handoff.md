## 2. 次にやること

### A. 画面のクオリティ向上（クライアントに見せられる水準へ）

現状は**機能検証用の見た目**で、そのまま人には見せられない。具体的な引っかかり:

| # | 箇所 | 問題 | 方向性 |
|---|---|---|---|
| A-1 | `/`（検索画面） | タイトルが「**検査UI**」、説明が「受け入れ試験用の最小クライアント。POST /api/v1/search を SSE で…」 | クライアント向けの言葉に。プロダクト名と用途を出す |
| A-2 | 利用者セレクタ | 「999 — 不明な利用者（401 期待）」等、**テスト用の選択肢が露出** | デモ用の3名だけに。異常系はテストへ寄せる |
| A-3 | 全画面 | 素の system font・最小CSS。ブランドも余白設計も無い | 一貫したビジュアル。**過剰装飾は不要**、業務システムとしての品位 |
| A-4 | URL | `?uid=1` が見えている | 見せ方の工夫（ヘッダのセレクタで完結させる等） |
| A-5 | `/scenarios`・`/audit` | 開発者向けの体裁 | デモ導線から外すか、見せるなら整える |
| A-6 | 一覧 | 要約が長いと横に伸びる。空状態・読み込み中の配慮が薄い | 情報設計（優先度の可視化・状態の見せ方） |

**判断が要る点**: 「デモ用の入口ページ」を作るか。
現状 `/` は検索画面。クライアントに見せるなら、3大課題と画面への導線をまとめた
トップがあると流れを作りやすい。**次セッションで CEO に確認すること。**

### B. 面接資料の充実

既存（`docs/handbook/`）:
- `interview-talking-points.html` / `.pdf` — **LLM評価ハーネスを実APIで走らせ切るまで**の話。完成度が高く、体裁の手本になる
- `interview-qa-script.html` / `.pdf` — 想定問答
- `devlog-2026-07-uat.md` — **今回の生ログ（前セッションで作成）。A・Bの一次資料はこれ**

やること: devlog を素材に、**今回のエピソード**を既存と同じ品質の資料に仕立てる。
既存2本と重複させず、追加するか統合するかは中身を読んでから決める。

**語れる素材（すべて実測・実ログ）**:
1. **順序の誤りと是正** — インフラ先行で指摘を受け凍結。「動くものを検証してから土台」
2. **認可の穴** — 品質管理部が月次を承認できた。しかも**既存テストが誤った振る舞いを固定していた**
3. **計測ミスの訂正** — 「検索3秒未達」は測る対象の誤り。要件は「3秒以内（**LLM生成部はストリーミング表示**）」。
   分解すると 検索128〜746ms / **初回表示880〜2,302ms** / 生成完了2,964〜4,519ms で**元から達成**
4. **テストダブルの盲点** — PDFの日本語が全部豆腐(□)。`FakePdfRenderer` で差し替えた経路は
   一度も実行されていなかった。「200が返る」で満足していた
5. **テスト4層の設計** — コストと頻度で分けた（課金ゼロで毎回 / 実APIでたまに）
6. **プロンプトインジェクション防御が実データで作動** — 報告書に仕込まれた指示をAIが明示的に拒否
7. **数値ハルシネーション対策** — 件数はSQL確定値、AIは文章だけ。実データで完全一致を確認
8. **AIが正直に「該当なし」と答えた。それでもデモは失敗だった** — データ設計の見落とし

### C. プレゼン用カンペ

**想定**: クライアント／面接官の前で画面を操作しながら説明する。

- **尺**: 何分想定かを CEO に確認（3分／5分／15分で構成が変わる）
- **必要なもの**: 画面遷移の順番、各画面で言うこと、詰まったときの逃げ道、想定質問への返し
- **軸**: 依頼主の3大課題 →「確認負荷」「事例の属人化」「月次作成の負荷」に対応する画面を順に見せる
- **見せ場**: ①未分類キューで人が直す改善ループ ②月次が数十秒で下書きまで出る
  ③検索が根拠リンク付きで答える ④権限で見える範囲が変わる
- **触れておきたい弱み**（誠実さの担保）: 認証はSSO未実装（`?uid=` の抽象点）、
  AWS未構築、現場スタッフ向け画面はスコープ外
