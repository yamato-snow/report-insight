# pgvector ベンチマーク（ADR-001 の実測裏取り）

[ADR-001](../../../docs/adr/ADR-001-vector-store.md) で「ベクトル100万件超 or 検索 p95 > 1秒 で
OpenSearch 移行を再検討」という閾値を決めた。本ベンチはその閾値までの余裕を、
本システム相当のスキーマ（`report_chunks`・`vector(1024)`・HNSW・メタデータフィルタ）で実測するもの。

- スクリプト: [bench.py](bench.py)
- 実測結果（生データ）: [bench_results.json](bench_results.json)

## 測定環境

| 項目 | 値 |
|---|---|
| マシン | Apple M1（8コア）/ 16GB RAM / SSD |
| PostgreSQL | 17.10（macOS ネイティブ、Docker 不使用※）+ pgvector 0.8.5 |
| ベクトル | `vector(1024)`・コサイン距離 |
| インデックス | HNSW（m=16, ef_construction=64 = pgvector デフォルト） |
| 検索 | top-10、各パターン 100 クエリ（ウォームアップ 10 回別途）の p50 / p95 |
| データ | 合成ランダムベクトル（正規化済み）＋物件ID・分類・報告日のメタデータ |

※ Docker Desktop(VM) 上で回すと大量 COPY で VM ディスクイメージが膨張しホスト容量を
圧迫したため、ネイティブ PostgreSQL（テーブルスペースは外付けSSD）に切り替えた。

## 実行方法

ローカル専用の PostgreSQL（例: ポート 55432、DB `bench`）を用意し、`bench.py` の `DSN` を合わせる。

```bash
# pgvector 拡張が入った PostgreSQL 17 が起動している前提
uv run --with numpy --with "psycopg[binary]" --with pgvector python bench.py
```

15万件 → 追い足しで50万件の順に、投入・HNSW構築・検索レイテンシを測って
`bench_results.json` に書き出す。50万件のHNSW構築に17分程度かかる。

## 結果サマリ

| 規模 | COPY投入 | HNSW構築 | インデックス | テーブル | top-10 p50 / p95 | フィルタ併用 p50 / p95 |
|---|---|---|---|---|---|---|
| 15万件 | 約10秒 | 約4.8分 | 1.17 GB | 0.80 GB | 13.4 / 29.5 ms | 41.7 / 49.0 ms |
| 50万件 | 約44秒（累計） | 約16.9分 | 3.91 GB | 2.67 GB | 29.5 / 36.6 ms | 250.5 / 283.2 ms |

- 最悪ケース（50万件・フィルタ併用 p95 283ms）でも ADR-001 の移行閾値 1秒 の約 1/3.5
- フィルタ併用は 50万件で約5倍に劣化（HNSW が後段フィルタで候補を再探索するため）。それでも閾値内
- 100万件では HNSW グラフ（推定 約8GB）が `maintenance_work_mem` に載らないとディスク構築
  モードに落ち、構築時間が桁で悪化する。「検索が遅くなる」前に「構築メモリの増強」が先に来る

## 注意（数値の読み方）

- 合成ランダムベクトルによる**参考値**。実データの埋め込みは分布が偏るため、絶対値は実運用と一致しない
- **recall@10 は載せていない**。高次元のランダムベクトルは近傍距離が縮退し（最近傍と遠方の距離差が
  小さくなり）、recall が実運用を代表しない値になるため。`bench.py` の `run_recall` は
  実データ埋め込みで測る場合のために残してある
